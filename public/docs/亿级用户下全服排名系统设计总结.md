# 亿级用户下全服排名系统设计总结

## 背景

项目需求需要向玩家展示"超越了全服 X% 的玩家"这一排名百分比。排名维度是多维的：优先比较收藏家等级，其次是典藏图鉴进度，最后是普通图鉴进度。

面向亿级用户量，这个需求在精度、性能、空间、扩展性之间需要反复权衡。下面按方案演进顺序梳理设计思路。

---

## 方案一：Redis ZSet

最直觉的方案——把每个玩家的综合分数直接写进 Redis Sorted Set，查排名时用 `ZRANK` 拿到绝对名次，再除以总人数换算为百分比。

**写入路径**：每次玩家数据更新，调用 `ZADD`，时间复杂度 O(log n)。

**问题**：

- 按每条记录 23 字节估算，1 亿用户需要约 2.3 GB 内存，直接占用过大。
- 亿级规模下 O(log n) 的写入压力不可忽视。
- 最危险的是 **BigKey 问题**——全量玩家数据都在同一个 Key 下，任何操作都可能造成 Redis 阻塞。

**结论**：只适合百万级以下用户，直接排除。

---

## 方案二：分段 key + 预计算

**核心思路**：不存每个玩家的排名，改为统计各分桶内的玩家数量分布，再通过分布估算百分比。

### Redis Key 设计

分桶 Key 结构为 `collector_rank_stat`（Hash），Field 格式：`{level}:{premium_base}:{normal_base}`，Value 为该分桶的玩家计数。

```
HSET collector_rank_stat "05:00000:01000" "1234"   // 5级，典藏0，普通进度1000-1999
HSET collector_rank_stat "06:01000:08000" "5678"   // 6级，典藏1000-1999，普通8000-8999
```

同时维护一张 `collector_rank_table`，每 5 分钟定时计算出每个分桶的排名区间（min% ～ max%）写入 Redis。

### 查询与估算

以某玩家 6 级、典藏进度 1500、普通进度 8500 为例，命中分桶 `06:01000:08000`，该桶排名区间为 5.2% ～ 6.8%：

1. **归一化进度**
   - 典藏比例 = (1500 - 1000) / (1999 - 1000) ≈ 0.5
   - 普通比例 = (8500 - 8000) / (8999 - 8000) ≈ 0.5
2. **权重计算**（假设典藏:普通 = 6:4）
   - 综合进度 = (0.5 × 6/4 + 0.5 × 4/6) / 2 ≈ 0.54
3. **估算排名**
   - 估算排名 = 5.2 + (6.8 - 5.2) × 0.54 ≈ 6.06%

**写入路径**：玩家数据更新时 `HINCRBY` 对应分桶计数，时间复杂度 O(1)。空间从 2.3 GB 压缩到约 23 MB。

**问题**：扩展性差。一旦策划调整排名维度（如新增满级图鉴数量），或图鉴等级/进度上限提升，分桶的 Field 数量会指数级增长，且需要重新扫盘迁移数据。

---

## 方案三：分段 score

**核心思路**：将多维度压成一个综合分数，再对分数空间做均等分桶。

### Score 设计

```
total_score = level × 1_000_000 + premium_progress × 100 + normal_progress × 1
```

权重系数保证高优先级维度不会被低维度"追上"，排序语义与原始多维排序完全等价。

### 查询与估算

以玩家进度 820、分桶步长 100 为例，命中分桶 `[800, 900)`：

```
估算排名 ≈ 该桶玩家数 / 总玩家数 × (820 - 800) / (900 - 800)
```

Redis 只需存约 1000 个分段计数，空间降至 **4 KB** 量级。

**写入路径**：玩家数据更新时 `HINCRBY` 对应分数段，时间复杂度 O(1)。

**问题**：如果需要横向扩展排名维度，调整权重后可能需要重新分段；分桶步长的选取也影响精度，需要提前规划。

---

## 三方案对比

| 方案 | 写入路径 | 查询路径 | 空间占用 | 扩展性 |
|------|----------|----------|----------|--------|
| Redis ZSet | O(log n) | O(log n) | 2.3 GB | 差（BigKey） |
| 分段 key + 预计算 | O(1) | O(1) | 23 MB | 一般 |
| 分段 score | O(1) | O(1) | 4 KB | 较好 |

三个方案都有一个共同问题没有解决：**策划修改排名规则时怎么办？** 无论是分桶设计还是权重系数，规则一变都意味着重新设计数据结构。

---

## 方案四（最终落地）：t-digest + PostgreSQL + 三级缓存

### 设计思路转变

把"实时维护分布"改为"定时离线重建分布"，用 **t-digest** 这一概率数据结构来表示全服分数分布。

t-digest 以极小的内存（几 KB 到几十 KB）维护数据流的分位数估计，对尾部数据（极高分/极低分玩家）精度更高，对中段玩家有合理的误差容忍——正好契合排行榜的实际需求：头部玩家需要精确区分，大量中段玩家允许有百分之几的误差。

此外，由于分数分布是离线重建的，排名规则改变时只需调整 score 的计算逻辑，重新跑一次全量计算即可，无需修改 Redis 数据结构。

### 整体架构

```
玩家分数更新 ──→ PostgreSQL（原始数据）
                      │
              定时任务（全量扫描）
                      │
              t-digest 计算分布
                      │
              序列化写入 Redis (L2)
                      │
              热加载到内存 (L1)
                      │
              GameSvr 直接查询 L1
```

### 三级缓存

| 层级 | 数据源 | 加载时间 | 大小 |
|------|--------|----------|------|
| L3 PostgreSQL | 原始数据（1 亿行） | ~21s（冷启动全量扫描） | ~6.7 GB |
| L2 Redis | 序列化后的 digest | ~39ms | 2.71 MB |
| L1 内存 | 反序列化后的 t-digest 对象 | 指针切换 1.2µs | < 5 MB |

**双缓冲设计**：后台异步重建新的 t-digest 对象，完成后原子替换 L1 指针。切换期间的查询无需加锁，阻塞时间仅 1.2µs，对业务完全透明。

**启动流程**：
- 冷启动（无 Redis 缓存）：L3 → L2 → L1，总耗时约 21.4s
- 热启动（有 Redis 缓存）：L2 → L1，总耗时约 200ms
- 运行中：查询直接命中 L1，定时后台刷新

### 写入与查询性能

**写入路径**：玩家操作直接落 PostgreSQL，对 GameSvr 无额外延迟，写入完全异步。

**查询性能**（压测数据，测试规模：1 亿玩家，分数范围 0 ～ 100000）：

| 场景 | 结果 | 说明 |
|------|------|------|
| 单次 L1 查询 | 137µs | 内存直接读取 |
| 并发查询 QPS | 195 万 | 10000 次并发 |
| 缓存切换阻塞 | 1.2µs | 双缓冲指针切换 |
| 读写混合 QPS | 7.1 万 | 50 读 + 50 写 + 频繁刷新 |

读写混合场景下吞吐下降主要来自刷新时的写锁竞争。对于排行榜这种读多写少（刷新周期分钟级）的场景，7.1 万的混合 QPS 远超实际需求。

---

## 关于 Flink 方案的取舍

讨论过程中也评估了用 Flink 流式增量计算的方案。两者的核心差异在于**实时性和架构复杂度的权衡**：

| 维度 | 定时全量计算 | Flink 流式增量 |
|------|-------------|--------------|
| 实时性 | 分钟/小时级 | 秒级 |
| 架构复杂度 | 低（Cron + 离线任务） | 高（Flink 集群 + Kafka + checkpoint） |
| 百分比精度 | 精确（全量扫描） | 精确（需处理乱序事件） |
| 容错 | 失败重跑 | exactly-once checkpoint |
| 维护成本 | 低 | 高 |

对于图鉴排名场景，玩家不需要秒级感知自己的排名变化（图鉴升级本身是低频操作），定时全量方案的实时性完全满足需求，没有必要引入 Flink 的额外复杂度和运维成本。

---

## 总结

这个排名系统的演进路径代表了一类典型的工程决策过程：

1. 从最直接的方案（ZSet）出发，识别出**空间和 BigKey** 是核心瓶颈。
2. 引入分桶统计的思路，将空间从 GB 级压到 MB 级，但暴露了**扩展性**问题。
3. 用综合 score 替代多维分段，进一步压缩空间，并提升了扩展灵活度。
4. 最终引入 t-digest 彻底解决了**规则变更**的问题，并通过三级缓存把查询性能推到极致。

最终方案的核心设计原则：**用离线计算换实时复杂度，用多级缓存隔离存储层延迟，用概率数据结构平衡精度与空间**。在亿级用户、低频写入、高频读取的场景下，这套思路是稳健且可维护的。
