为什么要无状态？

游戏服务器的早期形态几乎都是有状态的——玩家连上某台机器，这台机器就记住了他的一切。这在用户量小的时候毫无问题，但一旦要扩容，麻烦就来了：你不能随意把玩家切换到另一台机器，因为状态只在当前机器上。

无状态设计的核心思路很简单：服务器本身不持有任何玩家状态，所有状态都外化到存储层。每次请求都是独立的，任何一台 server 都可以处理任何一个玩家的请求。

💡 无状态不是说"没有状态"，而是说"状态不在服务器内存里"。

架构设计

我们的 gamesvr 是用 Go 写的，核心思路是：

  • 每个请求进来，先从 TcaplusDB 读取玩家数据到内存
  • 处理逻辑，修改数据
  • 写回 TcaplusDB，释放内存

这样每台 gamesvr 都是完全等价的，负载均衡器可以把请求打到任意一台。

TcaplusDB 的角色

TcaplusDB 是腾讯自研的 NoSQL 数据库，针对游戏场景做了很多优化。它支持 Generic 表、List 表和 SortList 表三种类型，分别对应不同的数据结构需求。

// 读取玩家背包数据示例
req := &tcaplusdb.GetRequest{
    TableName: "player_bag",
    Key: map[string]interface{}{
        "player_id": playerID,
    },
}
resp, err := client.Get(ctx, req)
if err != nil {
    return nil, fmt.Errorf("get player bag: %w", err)
}

并发安全：同一玩家的请求怎么处理？

无状态设计有一个经典问题：如果同一个玩家同时发了两个请求，两台不同的服务器都去读他的数据，都做了修改，然后都去写，怎么办？

我们用的是乐观锁方案——TcaplusDB 支持版本号，每次写入时带上读到的版本号，如果版本号不匹配（说明被别人改过了），写入会失败，然后重试。

总结

无状态化是游戏服务器走向大规模的必经之路。它的代价是多了一次存储 IO，但换来的是近乎无限的水平扩展能力。在我们的实践中，P99 延迟控制在 20ms 以内，完全满足游戏的实时性需求。
