问题规模

排行榜是游戏里最常见的功能之一，但当用户量到了亿级，它就变成了一个有趣的工程挑战。

需求很简单：给定一个玩家的战力值，实时返回他的全服排名。读写比大概是 100:1，对延迟非常敏感（P99 要求 50ms 以内）。

为什么 Redis ZSET 不够用？

最直觉的方案是 Redis 的 Sorted Set。ZRANK 命令可以返回排名，时间复杂度 O(log N)。

但当 N = 1 亿时，一个 ZSET 的内存占用在 10-20GB 量级，单个 Redis 实例扛不住，分片又带来了跨片聚合的问题。

分段统计方案

核心思路：把战力值分成若干区间（bucket），维护每个区间的人数统计。查询排名时，先查比自己战力高的所有区间的人数之和，再加上同区间内比自己高的人数。

// 区间划分：按战力值分成 1000 个 bucket
const BucketCount = 1000
const MaxPower = 10_000_000

func getBucket(power int64) int {
    return int(power * BucketCount / MaxPower)
}

// 查询排名
func GetRank(ctx context.Context, playerPower int64) (int64, error) {
    bucket := getBucket(playerPower)

    // 1. 所有更高 bucket 的人数之和（从预计算表读，O(1)）
    higherCount, _ := getHigherBucketsCount(ctx, bucket)

    // 2. 同 bucket 内比我高的人数（小 ZSET，用 Redis）
    sameCount, _ := rdb.ZCount(ctx,
        fmt.Sprintf("rank:bucket:%d", bucket),
        fmt.Sprintf("(%d", playerPower), "+inf",
    ).Result()

    return higherCount + sameCount + 1, nil
}

预计算表

每个 bucket 的累积人数预先计算并缓存。玩家战力变化时，只需要更新对应 bucket 的计数和两个小的 ZSET（旧 bucket 删除，新 bucket 插入）。

预计算表的更新频率：我们选择了准实时更新——每次战力变化时异步更新，延迟约 100ms，对排行榜展示完全可以接受。

效果

  • 内存占用：从 15GB 降到约 200MB（1000 个小 ZSET，每个平均 10 万人）
  • 查询延迟：P99 从 200ms 降到 8ms
  • 写入 QPS：支持 5 万/秒的战力变更
分而治之是解决大规模问题的通用思路。排行榜这个场景里，把一个巨大的有序集合切成很多小的，再加上预计算，问题就变得可控了。
