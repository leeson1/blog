问题背景

我们的视频处理系统基于 NVIDIA DeepStream，同时跑多路 RTSP 摄像头流，每路流在 GPU 上完成解码、推理、后处理的全流程。系统运行一段时间后，偶发性地出现图像撕裂和推理结果异常。

排查了两天，最终定位到 CUDA 内存对齐问题。

什么是内存对齐？

GPU 访问内存时有一个重要概念：内存对齐（Memory Alignment）。当内存地址满足特定字节对齐要求时，GPU 可以通过合并内存访问（Coalesced Memory Access）大幅提升带宽利用率。

CUDA 要求全局内存访问以 32/64/128 字节为单位对齐，否则会触发多次内存事务，性能下降，严重时会读到错误数据。

根本原因

我们在做图像拼接时，用 OpenCV 的 GpuMat 分配了一块显存，然后直接把这个指针传给了 DeepStream 的 buffer。问题是 GpuMat 内部按行对齐，每行末尾可能有 padding，而 DeepStream 假设数据是连续的。

// 错误做法：直接用 GpuMat 的原始指针
cv::cuda::GpuMat frame;
// ...
NvBufSurface* surface = ...;
cudaMemcpy(surface->surfaceList[0].dataPtr,
           frame.data,           // ← 这里有问题！
           width * height * 3,
           cudaMemcpyDeviceToDevice);

正确做法是使用 cudaMemcpy2D，显式指定源和目标的步长（pitch/step）：

// 正确做法：用 cudaMemcpy2D 处理 pitch
cudaMemcpy2D(
    surface->surfaceList[0].dataPtr,
    surface->surfaceList[0].pitch,  // 目标 pitch
    frame.data,
    frame.step,                      // 源 step（含 padding）
    width * 3,                       // 每行实际数据字节数
    height,
    cudaMemcpyDeviceToDevice
);

验证与修复效果

修复后运行了 72 小时压力测试，图像撕裂问题消失，推理准确率恢复正常。同时我们用 Nsight Systems 做了 profiling，内存事务数量下降了约 40%，说明对齐修复确实改善了访问效率。

这个坑的教训：在 CUDA 开发中，永远不要假设显存指针是连续的，尤其是来自第三方库的分配结果。
